# N6 任务与调度 面试 Flashcard

> 主题:任务模型、xTaskCreate、抢占式调度规则、两套优先级体系。

---

## 第一部分:纯面试(通用知识,不绑项目)

### A1. FreeRTOS 的任务函数为什么必须写成 for(;;) 永不返回?如果 return 会怎样?

**答**:任务是被调度器通过"伪造的初始栈帧 + 切换"跳进去的,不是被某个上层函数普通调用的,所以它没有有意义的"返回地址"可回。任务的生命模型里只有运行/就绪/阻塞/挂起/删除,**没有"自然结束"**。若 return,CPU 会跳到伪造栈帧里预填的返回地址 —— FreeRTOS 把它填成一个兜底函数 prvTaskExitError(一个断言 + 死循环),程序卡死在那;若该地址无效则直接 HardFault。

**怎么答出深度**:能指出这个返回地址在 port 层 pxPortInitialiseStack 里被填成 portTASK_RETURN_ADDRESS(默认 prvTaskExitError)。要"有管理地结束"任务,不能 return,要调 vTaskDelete(NULL),它会回收任务的栈和 TCB。讲清"return 是脱缰,vTaskDelete 是有管理退出"的区别是加分点。

**措辞陷阱**:别说"return 会回到 main"。任务和 main 不是调用关系,任务之间是平级、被调度器轮流唤醒的独立执行流,没有"上层栈"可回。

---

### A2. xTaskCreate 的几个参数分别是什么?有什么单位陷阱?

**答**:六个参数 —— 任务函数指针、任务名(字符串,调试用)、栈深度、传给任务的参数 pvParameters、任务优先级、任务句柄(输出,不需要填 NULL)。返回 pdPASS 成功,失败最常见是堆不足。

**怎么答出深度**:点出栈深度的单位陷阱 —— 原版(vanilla)FreeRTOS 单位是**字(word,4 字节)**,不是字节。ESP-IDF 改成了字节,照搬 ESP 文档会错。能讲"栈给小了运行时溢出 HardFault、给大了浪费 RAM,可用 uxTaskGetStackHighWaterMark 实测调整",显出工程意识。

**措辞陷阱**:configMINIMAL_STACK_SIZE 不是"所有任务的默认栈",它是**空闲任务的最小栈基准值**。每个任务的栈要在 xTaskCreate 里逐个指定,因为不同任务栈需求差别大(只翻 GPIO 的 vs 调 snprintf+浮点的)。

---

### A3. 抢占式调度的根本原则是什么?同优先级任务怎么处理?

**答**:根本原则 —— 任何时刻 CPU 必须执行**当前所有就绪任务中优先级最高的那个**。高优先级任务一旦就绪,立即抢占正在运行的低优先级任务。

**怎么答出深度**:抢占判定是**严格大于**:只有就绪任务优先级**严格高于**当前任务才抢占,**相等不抢占**。同优先级的多个就绪任务靠两种方式交替:① 当前任务主动让出(vTaskDelay/阻塞);② 时间片轮转(time slicing,默认开启,每个节拍在同级就绪任务间轮换)。能说出"严格大于、相等不抢"这个细节是高频加分点。

**措辞陷阱**:很多人误以为"同优先级也会互相抢占"。不会。同级靠主动让出或时间片,不靠抢占。

---

### A4. 任务优先级和中断优先级是一回事吗?

**答**:不是,是**两套完全独立的体系**。
- 任务优先级:xTaskCreate 第五参数,范围 0 ~ configMAX_PRIORITIES-1,**数字越大优先级越高**,归 FreeRTOS 软件调度器管,纯软件概念。
- 中断优先级:NVIC 优先级,范围如 0~15,**数字越小优先级越高**,归 Cortex-M 硬件 NVIC 管,决定中断间抢占和能否调 API。

**怎么答出深度**:两套方向相反(任务大为高、中断小为高),互不干涉。一个任务的优先级和一个中断的优先级之间没有可比性 —— 中断永远能打断任务(中断是硬件层面,任务是软件层面)。能把这两套清晰劈开,是初学者最容易混、面试官最爱考的点。

**措辞陷阱**:"configMAX_SYSCALL_INTERRUPT_PRIORITY ≥ 5 这条线"约束的是**中断优先级**,和任务优先级填几毫无关系。任务优先级填 1 是合法的,它不受那条线管。

---

### A5. 任务优先级 0 有什么讲究?

**答**:0 是惯例留给**空闲任务(Idle Task)**的级别(tskIDLE_PRIORITY = 0)。空闲任务是 FreeRTOS 在调度器启动时自动创建的,当没有任何用户任务就绪时它运行(负责回收被删除任务的内存等)。用户任务想跑得比空闲任务勤,优先级填 ≥1。

**怎么答出深度**:能提到空闲任务的作用(回收 vTaskDelete 的资源、可挂 idle hook 做低功耗),并说明"任何系统始终至少有一个就绪任务(空闲任务),所以 CPU 永远有事可做、调度器不会无任务可选"。

## 第二部分:项目挂钩(我 N6 实际怎么做的)

### P1. 你的 N6 多任务系统有哪些任务?优先级怎么定的?

我从单任务 LED 闪烁起步,逐步加到:vLedTask(优先级 1,翻转 PC13 + 上报状态)、vCmdTask(优先级 2,等串口命令信号量后响应)。vCmdTask 给得比 vLedTask 高,是故意的 —— 这样串口中断唤醒 vCmdTask 时,它能抢占正在跑的 vLedTask,让我能验证"高优任务被中断唤醒后立即抢占"以及 portYIELD_FROM_ISR 真正触发切换。

### P2. 你怎么亲眼验证"并发"的?和裸机有什么区别?

单任务 LED 闪烁其实和裸机 while(1)+HAL_Delay 看不出区别。我加第二个任务(串口打印)后才看到真并发:LED 按 500ms 闪、串口按自己节奏刷,互不阻塞。机制是两个任务靠 vTaskDelay/阻塞各自让出 CPU、轮流跑 —— LED 睡觉时 CPU 跑去打印,打印任务睡觉时 CPU 回来闪灯。这是裸机单条执行流永远做不到的:裸机里一个阻塞延时会锁死整条流。

### P3. 你被问过"为什么 HAL_Delay 在多任务下是灾难"吗?

会。HAL_Delay 是忙等(busy-wait):内部死循环空转读 HAL_GetTick,不交出 CPU。多任务下,一个任务调 HAL_Delay(500),CPU 被它死占 500ms,调度器没机会切到别的任务,并发退化成串行。而 vTaskDelay 是阻塞延时:告诉内核"我睡 500ms,这期间把 CPU 给别人",主动让出并触发调度。差别本质是"占着 CPU 空转" vs "让出 CPU 去干活"。

### P4. 同优先级的 LED 和 UART 任务,UART 那 2.2ms 的阻塞发送会拖累 LED 吗?

基本不会。LED 任务翻转完立刻 vTaskDelay(500) 进阻塞态,这 500ms 它根本不需要 CPU,UART 那 2.2ms 忙等绝大概率落在这段空窗里,不构成抢占。唯一极小概率的影响:LED 睡眠刚到期该翻转时,UART 正卡在 2.2ms 忙等中间 —— 因为同级不抢占,LED 这次翻转最多被推迟 2.2ms,500ms 周期里肉眼不可见。若把 LED 优先级设得比 UART 高,连这 2.2ms 都没有,因为高优 LED 到期会立即抢占忙等中的 UART。

### P5. 栈大小你怎么分配的?为什么不统一?

按需分配。vLedTask 起初只翻 GPIO,给 configMINIMAL_STACK_SIZE(512 字节)够;后来它加了 char buf[32] + snprintf,栈需求涨了,改成 *2。vCmdTask 有 buf + snprintf 从一开始就给 *2。不统一是因为 snprintf 调用层次深、用栈多,而 configMINIMAL_STACK_SIZE 只是最小基准、不是默认值。这也是 F103C8T6 只有 20KB RAM 下必须精算的原因。
