## 一、协议本质

**Q1：UART、I2C、单总线三者在"怎么确定每一位"上的根本区别？**
A：
- UART：无时钟线，靠**预先约定的波特率**+各自本地时钟，在固定时刻**采样电平的高低**。
- I2C：有独立时钟线 SCL，由 SCL 划分每一位边界。
- 单总线：只有一根 DATA，既传数据又承担定时，靠**测量电平持续的时长**编码 0/1。
- 一句话：**UART 用"固定时刻采电平"传信息，DHT11 用"测量电平持续时长"传信息。前者硬件能代劳，后者必须 CPU 用微秒延时去量，所以只能软件模拟。**

**Q2：为什么 DHT11 时序必须软件模拟，不能交给硬件外设？**
A：STM32F1 没有专门的"单总线外设"。每一段高低电平、各持续多少微秒，都要用裸 GPIO 手动拉电平 + 手动微秒延时 + 手动读电平完成。它是 N1（GPIO输出）+ N2（GPIO输入）+ 精确微秒延时三者的综合实战。

## 二、开漏与上拉

**Q3：单总线的 DATA 引脚为什么必须用开漏输出 + 上拉，而不能用推挽？**
A：单总线一根线收发轮流（半双工），同一时刻只能有一个设备驱动它。
- 开漏：只主动拉低，输出高时"松手"（下管断开），靠外部上拉回高，永不主动驱动高。
- 多个开漏挂同一根线绝不冲突——这叫**线与（Wired-AND）**：谁拉低算谁，没人拉低就被上拉拉高。
- 推挽：输出高时上管硬接 VCC。若一方推挽输出高、另一方拉低 → 强驱动对打 → **总线竞争（bus contention）**，电平变中间值（数据全错）+ 大电流直灌 → 烧 IO 风险。
- **与 I2C 的 SDA/SCL 开漏+上拉同源，原理完全一样。**

**Q4：成品 DHT11 模块和裸件在接线上的关键差异？**
A：DATA 必须有 4.7kΩ 上拉。成品模块（3 脚）板载已焊上拉，直插 VCC/GND/DATA；裸件（4 脚）须自己在 DATA-VCC 间接上拉电阻，否则总线浮空，通信必失败且现象与代码 bug 一模一样。

## 三、方向切换

**Q5：DHT11 通信中 GPIO 方向怎么切？"松手"在代码里具体怎么实现？**
A：节奏是 `配成输出 → 拉低18ms → 切成输入`。
- "释放总线/松手"在 STM32 上**直接把 GPIO 切回输入模式**实现：引脚不再驱动，外部上拉把线拉回高（=松手），且输入模式正好能 `HAL_GPIO_ReadPin` 读 DHT11 电平。
- **一个动作（切输入）同时完成"释放总线"+"准备接收"两件事。**
- 这是 N5 区别于 N1~N4 的独门操作：N1~N4 引脚方向开机配好就不动，N5 第一次要"边跑边改方向"。

## 四、微秒延时 DWT

**Q6：为什么 DHT11 延时不能用 HAL_Delay，要用 DWT？**
A：0 的高电平 26~28μs、1 的高电平 70μs，要分辨几十μs 的差。HAL_Delay 最小单位 1ms，在它眼里 26μs 和 70μs 都是"0ms"，无法分辨。必须用微秒级工具。

**Q7：DWT 是什么？为什么选它而不是改造 SysTick？**
A：DWT（Data Watchpoint and Trace）内核里带一个 CYCCNT（Cycle Counter）32位计数器，开启后 CPU 每过一个时钟周期 +1。72MHz 下 1μs=72 周期，天然亚微秒精度。
- 选 DWT 不选 SysTick：SysTick 已被 HAL 占用产生 1ms 系统节拍（HAL_GetTick 靠它），改它会干扰系统时基；DWT 是独立、闲置的计数器，不抢任何资源。
- 一句话：**DWT 是一个免费的、72MHz 的、谁都没用的硬件秒表。**

**Q8：DWT_Init 的三道开关顺序？**
A：
1. `CoreDebug->DEMCR |= CoreDebug_DEMCR_TRCENA_Msk;` —— 开 DWT 单元总电源（不开它 CYCCNT 设了也不动）
2. `DWT->CYCCNT = 0;` —— 计数器清零
3. `DWT->CTRL |= DWT_CTRL_CYCCNTENA_Msk;` —— 开 CYCCNT 计数开关

**Q9：delay_us 里为什么用无符号减法 `(DWT->CYCCNT - start)`？**
A：CYCCNT 32位约 59 秒回绕一次。用 uint32_t 无符号减法，即使中途溢出，`(now - start)` 结果依然正确（如 start=4294967290, now=10 → 无符号下得 16，非负数），不用为溢出写特殊处理。同类思想见 N3 环形缓冲 oldPos/newPos。
- 隐藏边界：`us * 72` 单次最大约 59 秒（2³²/72），对 DHT11 绰绰有余，但要知道上限存在。

## 五、读位技巧与时序

**Q10：DHT11 一次完整通信的四段时序？**
A：
1. 主机起始：STM32 拉低 ≥18ms，松手切输入
2. DHT11 响应：80μs 低 → 80μs 高（准备）
3. 传 40 位数据：每位 50μs 低 + 高电平（长短编码 0/1）
4. 结束：DHT11 拉低 50μs 后松手回空闲

**Q11：读一位时怎么区分 0 和 1？为什么不去精确测高电平时长？**
A：用"延时中间值，读一眼"的偷懒法：
- 每位以 50μs 低开始（0/1 共有、无信息）→ 先等低结束、高电平开始。
- 高电平开始后 `delay_us(40)`（40 卡在 28 和 70 中间），再读一眼电平：
  - 是 0（高~28μs）：40μs 时早已掉回低 → 读到低 → 判 0
  - 是 1（高 70μs）：40μs 时仍在高 → 读到高 → 判 1
- **把"测一段时长"简化成"延时固定值 + 单次采样"**，甩掉精确测量，抗抖动。
- 读完若是 1（仍在高电平），要再等它掉回低，对齐下一位。

**Q12：组装字节为什么用 `byte <<= 1; byte |= 1;`？**
A：每读一位先左移腾出最低位，再把新位填进最低位。8 次后最先读到的位被挤到最高位 → 实现"高位先出"（MSB first）。N8 封 CRC 帧同一招。

**Q13：DHT11 的 40 位数据结构和校验？**
A：40 位 = 8湿度整数 + 8湿度小数 + 8温度整数 + 8温度小数 + 8校验和，高位先出。校验和 = 前 4 字节相加结果的**末 8 位**。代码：`(uint8_t)(data[0]+data[1]+data[2]+data[3]) != data[4]`，强转 uint8_t 取末 8 位（相加可能 >255，进位必须丢弃）。DHT11 测整数，两小数字节通常为 0。

## 六、超时保护（命门）

**Q14：为什么单总线读取每处"等电平翻转"都必须带超时？**
A：若写成纯 `while(电平==高);` 死等不超时，一旦传感器没插好/坏了/上拉异常，总线永远是高，while 永远跳不出 → **整个程序卡死**。
- 加超时：最坏返回错误码，主循环继续跑、能打印"无响应"。这才是工业级健壮写法。
- 一句话：**单总线每处等翻转都必须带超时，否则传感器掉线会让 MCU 死循环卡死；超时后返回错误码让上层感知并恢复。**
- 实现：用 DWT 计时（不是循环计数法，后者靠主频估、不可移植），`(CYCCNT-start) > timeout*72` 即超时。

**Q15：DHT11_Read 三种返回值的意义？**
A：0=成功、1=无响应（查接线/供电）、2=校验错（数据传错，可重读）。让上层区分"传感器掉线"和"数据传错"分别处理。
- 实战印证：开机第一行 `DHT11 no response` 不是 bug——DHT11 上电需~1s 稳定，首读超时返回 1，但程序没卡死、继续跑，就绪后正常读出。**这正是超时保护价值的活体证明。**

## 七、两个实战排障坑（高频，面试可讲）

**Q16：【坑1】printf 重定向到串口，钩子函数因工具链而异？**
A：
- **Keil/ARMCC**：printf → `fputc`，重写 `fputc`。
- **GCC/newlib（CubeMX 默认）**：printf → `_write`（syscalls.c）→ `__io_putchar`，重写 `__io_putchar`。
- 用错钩子：函数编译进去了但 printf 根本不调用它（孤儿函数），**编译/链接都不报错**，运行时串口完全无输出，连开机提示都没有。
- GCC 工程正确写法：
  ```c
  int __io_putchar(int ch) {
      HAL_UART_Transmit(&huart1, (uint8_t*)&ch, 1, 0xFFFF);
      return ch;
  }
  ```
- 排查方法论：对比"能用的旧工程"（N3 直接用 HAL_UART_Transmit 发帧，没走 printf，躲过此坑）找唯一差异（N5 首次用 printf）→ 锁定 printf 钩子。
- 附加：newlib-nano（--specs=nano.specs）默认不支持 `%f` 浮点 printf，要打浮点需加链接选项 `-u _printf_float`。

**Q17：【坑2】串口 COM 端口号漂移？**
A：CH340/USB-TTL 模块每次换 USB 口或重插，Windows 可能分配**新的 COM 号**，旧号变成失效残留记录。现象：MobaXterm 连旧号（COM5）时端口"存在"但收不到任何数据，与"接线/配置错误"现象一样。
- 排查：拔插法（拔掉看哪个 COM 消失）或设备管理器确认当前真实号；usbipd list 看 STATE。
- loopback 自测：短接 USB-TTL 模块自己的 TXD-RXD，敲字符看回显——有回显=端口/模块/波特率都对，问题在 MCU 侧；无回显=问题在 PC 侧（端口/驱动/占用）。

**Q18：这套混合开发环境，ST-Link 和 USB-TTL 的归属规则？**
A：
- **ST-Link（烧录）attach 进 WSL**（OpenOCD 在 WSL 跑）：`usbipd attach --wsl --busid <id>`
- **USB-TTL 串口（看打印）留在 Windows**（MobaXterm 在 Windows 跑）
- 两个设备归属相反。若误把串口也 attach 到 WSL，Windows 侧 MobaXterm 看得到端口号但收不到数据。

## 八、N6 预埋隐患

**Q19：delay_us 是忙等待，到 FreeRTOS 阶段有什么风险？**
A：delay_us 是 busy-wait，CPU 100% 占住死等。裸机没问题。但 N6 上 FreeRTOS 后，若开了高频中断（如 N3 UART 中断），中断会插进 `delay_us(40)` 中间，导致那一眼读偏，校验错。读 DHT11 那段需临时关中断保护时序。N6 处理。
