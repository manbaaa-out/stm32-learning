## N6 · FreeRTOS 基础(面试为主)

### Part 1 · 面试

**Q1. 为什么用 RTOS,不用裸机超循环(while(1))?**

裸机超循环把所有任务排成一条 while(1) 轮流执行,病根在于**实时性无法保证**:只要其中一段是耗时或阻塞操作(阻塞读传感器、等串口),整条循环就被它卡住,别的任务再紧急也得排队等它跑完。要硬撑,就得手动把每个任务切成小块、配状态机轮询,代码很快变得难维护。

RTOS 的核心是**抢占式调度**:任务各自独立成栈、带优先级,调度器永远运行优先级最高的就绪任务,一旦更紧急的任务就绪(比如中断把它唤醒),立刻打断当前任务去跑它——紧急任务的响应延迟因此有上界,而不是取决于别人跑多久。附带还有:任务阻塞/延时时主动让出 CPU(不像裸机延时空转),以及队列、信号量这些现成的通信原语。

代价也要讲清:多了内核的 RAM/Flash 开销、每次切换的上下文保存恢复成本、以及调度本身的复杂度(优先级怎么分、共享资源怎么护)。换来的是**确定性的实时响应**,所以它适合"多个轻重缓急不同的任务并存"的场景,而不是越用越好。

**Q2. 任务有哪几个状态,转换条件是什么?**

四个:运行(Running)、就绪(Ready)、阻塞(Blocked)、挂起(Suspended)。
- **就绪 → 运行**:调度器从就绪任务里挑优先级最高的投入运行;被更高优先级任务抢占时退回就绪。
- **运行 → 阻塞**:任务主动等一件事——延时(vTaskDelay)、等队列来数据、等信号量——可带超时;等到/超时前完全不占 CPU。
- **阻塞 → 就绪**:等的事发生了,或超时到了。
- **挂起**:由 vTaskSuspend 显式进入,与阻塞的区别是它**不带等待条件、也不超时**,只能由 vTaskResume 唤醒,期间完全不参与调度。

要点(也是常被漏的那句):阻塞和挂起都**让出 CPU**——这正是 RTOS 比裸机省的地方。裸机里 HAL_Delay 那种是空转(CPU 在原地空跑),而任务阻塞期间 CPU 已经去跑别的任务了。

**Q3. 抢占式调度的规则?同优先级怎么办?**

一句话:**任何时刻运行的,一定是当前所有就绪任务里优先级最高的那个**。一旦有更高优先级任务变就绪(最常见是中断里 give 了它在等的信号量/队列),立刻抢占——不等当前任务跑完、也不等下一个 tick。

同优先级的多个就绪任务:在 `configUSE_TIME_SLICING=1` 时按系统节拍做时间片轮转,每拍轮一个;关掉时间片则当前任务一直跑到自己阻塞才换人。容易忽略的边界:抢占只发生在"出现更高优先级就绪任务"或"当前任务主动阻塞/让出"时,**同优先级之间不互相抢占**,只在 tick 边界轮转。

**Q4. 任务优先级按什么分配?**

判据是**响应的紧迫性(实时性要求),不是任务的"重要程度"**。哪个任务一旦该跑就必须尽快被响应、否则出问题,它才该高优先级。

最典型的反模式:把一个耗时但不紧急的后台活(数据统计、刷屏)设成高优先级——它一跑就长时间霸占 CPU,把真正紧急的任务(要及时应答的通信、要按时采的传感器)饿死。所以"重要"和"紧急"要分开:重要但不紧急的活,反而要放低、让它见缝插针地跑。

落到本项目:命令任务最高,因为网关有 500ms 应答时限、必须最快响应;TX 次之;周期性的采样上报最低,晚几毫秒无伤大雅。

**Q5. 任务栈大小怎么估?估错了会怎样?**

分四步答。

**① 先搞清栈上到底有什么。** 任务栈 = 函数调用最深那条路径上每层栈帧(入参、返回地址、被调者保存的寄存器、局部变量)的累加;吃栈大户是大数组/结构体局部变量、`printf`/`sprintf`(几百字节)、**软件浮点**(F103 是 Cortex-M3、没有硬件 FPU,浮点运算靠 libgcc 函数实现、走普通调用栈)、递归。一个常被答漏的关键点:Cortex-M 上**任务跑在 PSP、中断跑在 MSP**,所以中断 handler 的局部变量和**中断嵌套**吃的是**主栈(MSP)、不吃任务栈**;任务栈只需额外留**一个硬件自动压入的异常帧**(M3 是 8 个字 / 32 字节,即"任务正跑着被中断打断"那一下硬件压在 PSP 上的那帧;带 FPU 的 M4F 开惰性压栈才是 26 个字)。

**② 实测才靠谱——栈高水位线。** FreeRTOS 建任务时把栈填成 `0xA5`,`uxTaskGetStackHighWaterMark()` 扫这片填充被覆盖到哪,就知道任务**历史上用到的最深点**。做法:初版给宽裕(比如 ×3)→ **跑一遍最坏路径**(最深的命令分发 + 采样 + 组帧 + 浮点都走到)→ 读高水位 → 按"实际用量 + 20~30% 余量"裁。需开 `INCLUDE_uxTaskGetStackHighWaterMark`。

**③ 兜底两手。** `configCHECK_FOR_STACK_OVERFLOW=2`(方法 1 在任务切换时查 SP 是否越界,方法 2 再查栈尾那段 `0xA5` 还在不在)+ `vApplicationStackOverflowHook` 抓运行期溢出;编译期用 GCC `-fstack-usage` 出每个函数的栈帧大小,沿最深调用链求和估上界。

**④ 为什么较真——权衡。** 估小了:溢出踩坏相邻任务的栈或 TCB,表现为莫名 HardFault 或数据被悄悄改、极难定位;估大了:在 20KB RAM 上白吃内存、还可能挤垮别的任务和堆。所以策略是**宁可先大、实测高水位、再留余量往回收**。

**Q6. vTaskDelay 和 vTaskDelayUntil 区别?**

两个都是延时,区别在**计时的基准点**:
- vTaskDelay(N) 是**相对延时**——从"调用它那一刻"起睡 N 拍。于是一轮实际周期 = 这轮干活耗时 + N,干活耗时一波动,周期就跟着漂、还会越积越多。
- vTaskDelayUntil(&last, N) 是**绝对延时**——它睡到"上一次唤醒时刻 + N"这个绝对时间点,干活花了多久都从上次唤醒点往后算,**自动把干活耗时补偿掉**,周期严格等于 N。

所以凡是要**精确、不漂移的周期**(定时采样、固定心跳),必须用 vTaskDelayUntil。它有个**前提条件**要答出来:单轮干活耗时必须 < N,否则它"追不上"绝对栅格、会连着不延时地补跑。本项目采样上报就靠它卡 1s 栅格——采样那轮约 185ms,远小于 1s,所以稳。

**Q7. tick 是什么?上下文切换到底在哪做、怎么做?**

tick 是 FreeRTOS 的系统时基,由一个定时中断周期性产生——Cortex-M 上通常就是 **SysTick**,频率由 `configTICK_RATE_HZ` 定(常见 1000Hz = 1ms 一拍)。用途:给所有延时/超时计时,给同优先级任务做时间片轮转。每来一拍,内核检查有没有任务该从阻塞转就绪、要不要切换。

切换本身放在 **PendSV** 这个异常里做,而且 FreeRTOS 故意把它设成**最低优先级**——保证它只在所有别的中断都处理完后才执行,不会打断真正的 ISR。切换三步:把当前任务被调用者保存的寄存器(R4–R11 等)压进**它自己的栈**、把它的栈指针(PSP)存进 TCB;选出下一个任务;从那个任务的 TCB 取回栈指针、弹回它的寄存器。SysTick 里(或某个中断里)发现需要切换,就挂一个 PendSV 请求,等中断退出后由 PendSV 完成。

**Q8. FreeRTOS 的内存从哪来?几种堆怎么选?**

动态创建对象(xTaskCreate、xQueueCreate 等)的内存,都从 FreeRTOS **自己管理的一块堆**里出,大小由 `configTOTAL_HEAP_SIZE` 定,和 C 库的 malloc 堆是两码事。它提供 heap_1~5,差别在能不能释放、能不能合并碎片:heap_1 只分配不释放(最简单,适合任务一次建好就不删的系统);heap_2 能释放但不合并相邻空闲块(易碎片);**heap_4 能释放且合并相邻空闲块,最常用**;heap_5 是 heap_4 再支持多块不连续内存;heap_3 则是给标准库 malloc/free 套个线程安全壳。

要答出的实践点:动态创建会失败(堆不够时返回 NULL / pdFAIL),所以**创建后必须检返回值**;或者用静态创建(xTaskCreateStatic 等)把栈和控制块的内存自己以静态数组给出去、完全不走堆,占用编译期就定死、更可控,资源紧的产品常这么干。

### Part 2 · 项目

**Q. 你这节点的任务怎么划分、优先级怎么定、栈怎么给?**

三任务一中断,优先级严格"命令 > TX > 采样上报":
- **命令任务(3,最高)**:阻塞读 RX 流缓冲,拼帧后分发执行查询/设周期。给最高是因为网关有 500ms 应答时限,它必须最先被响应。
- **TX 任务(2)**:串口的**唯一写者**,从 TX 队列取帧写 UART,避免多任务同时写串口造成字节交错。
- **采样上报任务(1,最低)**:周期发心跳、按采样周期采数据上报,晚几毫秒无所谓;用 vTaskDelayUntil 卡 1s 绝对栅格防漂移。

栈给的是 configMINIMAL ×2(TX)、×3(采样、命令,因为它们要走 sample_all + 组帧 + 软件浮点、调用链更深);启动时三个 xTaskCreate 都检了返回值,堆不够当场进 Error_Handler、而不是静默创建失败。

## N7 · 任务间通信(面试为主)

### Part 1 · 面试

**Q1. 队列为什么是值拷贝,而不是传指针?**

FreeRTOS 队列传数据是把整块数据**拷进**队列的内部存储,接收方再从队列拷出一份到自己的变量。这么设计是为了**所有权清晰、生命周期安全**:发送方 push 完,它那个局部变量哪怕立刻出作用域被销毁,也不影响队列里那份拷贝;接收方拿到的是独立一份,两边互不干扰。

若改成传指针(只把地址放进队列),省了拷贝,但发送方就得保证"接收方取走并用完之前,被指向那块内存一直有效"——局部变量肯定不行(早释放了),只能用动态分配或静态缓冲并自己管释放时机,极易踩悬空指针。所以**小数据直接值拷贝**(几十字节的拷贝在低速场景可忽略);只有数据很大、拷贝代价高时,才用"队列里传指针 + 一套缓冲区所有权约定",那是拿复杂度换性能的权衡。

**Q2. 二值信号量和计数信号量,各用在什么场景?**

- **二值信号量**(只有"有/无"两态):用于**同步与事件通知**。最典型是中断和任务配合——中断里 give 一下表示"事件来了",任务平时 take 着阻塞、被唤醒后去处理。它表达的是"某件事发生了没有"。
- **计数信号量**(内部是个计数器):用于**对一类有限资源计数**或累计事件次数。比如有 N 个相同缓冲块,take 减一(占一个)、give 加一(还一个),计到 0 就让后来者阻塞等。

一句话区分:**二值管"事件发没发生",计数管"还剩几个"**。

**Q3. 互斥量和二值信号量看着都能"加锁",区别到底在哪?**

底层都能当"令牌"用,但互斥量为"保护共享资源"专门多了两点,这正是考点:
1. **优先级继承**:当高优先级任务被卡在某个低优先级任务持有的互斥量上时,内核会把那个**持锁的低优先级任务临时提升到高优先级**,让它尽快跑完临界区、放锁,以此压制优先级反转。二值信号量没有。
2. **所有权**:互斥量是"谁 take 谁 give",有明确持有者;二值信号量没有所有者,完全可以一个任务 give、另一个任务 take(这正是它适合做同步而非互斥的原因)。

所以判据很硬:**保护共享资源(同时只能一个人用)→ 互斥量**,要的就是那份优先级继承;**纯事件同步/通知(尤其涉及中断)→ 二值信号量**。反过来用——拿二值信号量去保护资源——就丢了优先级继承、埋下优先级反转的雷;拿互斥量去做 ISR 同步更是错的(下一条讲)。

**Q4. 什么是优先级反转?互斥量的优先级继承具体怎么解?**

场景要能当场讲出来:三个任务,优先级 高 H > 中 M > 低 L。L 先拿到锁 R;此时 H 就绪、也要 R,被阻塞、只能等 L 放;**偏偏这时 M 也就绪了,M 不碰 R,但优先级比 L 高,就把 L 抢占了**。结果 L 迟迟跑不完临界区、锁迟迟不放,H 就被一个跟它毫无资源关系的 M **间接地无限期拖住**——高优先级任务的实时性被中优先级任务毁了。NASA 火星探路者当年就栽在这上面(看门狗因此反复复位)。

**继承怎么解**:H 一阻塞在 L 持有的互斥量上,内核立刻把 **L 临时提到和 H 一样高**。这样 M 抢不动 L 了,L 得以尽快跑完临界区、放锁;放锁瞬间 L 的优先级**还原**,H 拿到锁继续。本质是"让持锁者临时借用等锁者里最高的优先级,把临界区尽快做完"。

要补的边界:继承只是**压制反转、缩短被拖延的时间,并不能消除阻塞**(H 该等的临界区还得等),所以临界区本身一定要短。另一条路是优先级天花板(进锁就提到预设最高级),开销更小,但要求预先知道天花板值。

**Q5. 中断里为什么必须用 FromISR 版 API?为什么不能在中断里用互斥量?**

普通版 API(xQueueSend、xSemaphoreTake 等)在队列满/空时会**让任务进入阻塞**、并可能触发切换,而中断**根本没有"任务上下文"可阻塞**——ISR 不是任务,你没法把它挂起来等、它也不该等。所以中断里必须用 `...FromISR` 版:它们**绝不阻塞**,满了直接返回失败,并通过出参告诉你"是否唤醒了更高优先级任务"(见下条),把是否切换的决定权交回给你在中断退出时处理。

互斥量更进一步**完全不能在中断里用**:它的核心价值——优先级继承——建立在"持有者是某个任务"之上,而中断没有任务身份,谈不上谁继承谁;何况 take 本身就是可能阻塞的操作。所以中断里要跟任务同步,只能用二值信号量的 GiveFromISR,不能用互斥量。

**Q6. portYIELD_FROM_ISR 到底解决什么问题?**

中断里用 FromISR 唤醒了一个任务,但它**未必能立刻跑**——得等中断返回、调度器重新挑。问题是:如果被唤醒的任务优先级比"刚被这个中断打断的那个任务"还高,你肯定希望中断一退出就立刻切到它,而不是先切回那个被打断的低优先级任务、傻等到下一拍才轮到高优先级任务。

机制:FromISR 的 API 有个出参 `pxHigherPriorityTaskWoken`,被唤醒任务优先级更高时它被置位;你在中断末尾把它传给 `portYIELD_FROM_ISR(woken)`,后者据此**在中断退出时立即触发一次上下文切换**,让高优先级任务马上跑。不调它,功能上不算错,但响应会白白延迟最多一个 tick——这对"中断驱动、要求低延迟"的场景(本项目串口收到命令要尽快处理)是实打实的损失。

**Q7. StreamBuffer 是什么?和队列怎么选?**

StreamBuffer 是 FreeRTOS 的**字节流**缓冲:写多少字节、读多少字节,不分"消息边界",适合传连续数据流(比如串口收到的原始字节)。它最大的特点是为**单写者单读者**场景做了优化——在这个约束下内部**不需要加锁**,开销很小。

队列是**定长消息**:每个元素大小固定、按条收发,且支持多读多写,但为此要付锁的开销。

判据:传"一条条结构化消息"(比如一帧已组好的 TxFrame)→ 队列;传"一串没固定边界的字节、且明确只有一个生产者一个消费者"→ StreamBuffer。本项目串口接收正是后者:RX 中断一个人写、命令任务一个人读出来喂解析器,所以用 StreamBuffer 而非队列。

**Q8. 临界区(taskENTER_CRITICAL)和互斥量,什么时候用哪个?**

两者都能护共享数据,但代价和粒度完全不同:
- **临界区**靠**关中断**(提升屏蔽优先级)实现,期间连中断都被挡、谁也抢不了。优点是开销极小、不阻塞;缺点是把整个系统的中断响应都停了,所以**只能用于极短的临界段**——几条指令、改个标志、读写一个多字节变量这种。
- **互斥量**护的是**可能较长、甚至会阻塞的临界段**(比如读一个要等几十毫秒的传感器):它只挡其他要这把锁的任务,期间中断照常响应、别的任务照常调度,还带优先级继承。

判据:**临界段极短、不含会阻塞的调用 → 临界区;临界段较长或里面有可能阻塞的调用 → 互斥量**。本项目里 sample_all 要读传感器(慢)、用互斥量;而 DHT11 那种微秒级时序的位读取段,则用临界区把那一小段裹起来、防止被打断打乱时序。

### Part 2 · 项目

**Q. 你这节点用了哪些任务间通信对象,各自为什么这么选?**

三个,各有讲究:
1. **TX 队列**(元素是 TxFrame_t):采样上报任务和命令任务两个生产者都往里投帧,TX 任务单独消费、负责实际写串口。用队列(值拷贝)而非共享缓冲,是配合"**串口唯一写者**"这条铁律——所有要发的帧先排进队列、由 TX 任务串行发出,杜绝多任务同时写 UART 造成字节交错。
2. **g_sensor_mutex**(互斥量):裹住 sample_all,保证同一时刻只一个任务读传感器。这里特意**选互斥量而非二值信号量,就是要优先级继承**——命令任务(高)和采样上报任务都会去采,万一采样任务持锁时命令任务来等,继承能避免它被拖久。
3. **RX 流缓冲**(StreamBuffer):接收中断是唯一写者、命令任务是唯一读者,单写单读、无需加锁。中断回调里用 xStreamBufferSendFromISR 灌字节,再用 portYIELD_FROM_ISR 让(更高优先级的)命令任务尽快被唤醒处理。

## N8 · 协议帧封装(项目为主)

### Part 1 · 面试

**Q1. 自定义串口帧协议,为什么要"帧头 + 长度 + 校验"这套结构?**

根子在于**串口是没有边界的字节流**:对端连续发来一串字节,接收方自己并不知道"一帧从哪开始、到哪结束"。这套结构就是来解决分帧和可靠性:
- **帧头**给出一帧起点。本协议用两字节 0xAA 0x55 而非一字节,是为降低误判:单字节帧头被 payload 里恰好同值的字节"假冒"的概率是 1/256,两字节降到 1/65536,低 256 倍。
- **长度(LEN)**告诉接收方这帧后面还有多少字节,据此知道收到哪算一帧完,而不是靠猜。
- **校验(CRC)**对抗传输中的位翻转(嵌入式现场电磁干扰是常态),收完一帧验一下,错了整帧丢。

三者合起来,让接收端状态机**永远不卡死**:任何一步出错(帧头不对、长度越界、CRC 不过)都退回"找帧头"的起点重新同步,最坏只丢当前这一帧,下一帧照常解。这是"标记流派"——靠固定标记分帧,区别于 Modbus RTU 那种靠字符间静默时间分帧的"时序流派"(后者在跑 Linux 用户态、保证不了字符级时序的接收端上并不可靠)。

**Q2. CRC16 为什么选 MODBUS?查表和现算怎么权衡?**

选 CRC16-MODBUS 主要因为它是**嵌入式串口领域的事实标准**(Modbus 生态、各种传感器/屏都在用),各端实现好对齐;且 CRC16 对几十到上千字节的短帧检错能力足够,开销也小。

查表 vs 现算是**空间换时间**的经典权衡:
- **查表法**预先算一张 256 项的表(512 字节),每处理一字节只需一次查表加异或、约十个时钟周期,**快**,适合高吞吐或 CPU 紧张,代价是那 512 字节。
- **逐位现算**不要表、省那 512 字节,但每字节循环 8 次移位,**慢一些**,适合资源极紧或低速。

本项目 ≤10Hz 低速、帧又小,两者都无所谓;我实际是**复用项目里既有的那份 crc16_update**(解析器、组帧、网关三处同一份),首要保证的是三端结果一致,而不是抠这点速度。

**Q3. 大小端在你的协议里怎么约定?为什么要明确约定?**

必须**两端事先把字节序定死**,否则同一个多字节字段,发端按一种顺序摆、收端按另一种顺序解,值就完全错位——这是跨设备协议最隐蔽的 bug。

本协议分两层:**帧级的 CRC 用小端**(低字节在前,CRC_LO 先于 CRC_HI),**payload 内部的多字节业务字段用大端**(网络字节序,如温度、光照值都高字节在前),上下行一致。把规则白纸黑字写进文档、再用测试向量验证,就是为了让发收两端不会在字节序上各执一词。

### Part 2 · 项目

**Q1. 完整讲一下你这套 M5 帧的格式。**

`0xAA 0x55 | LEN | TYPE | payload | CRC_LO | CRC_HI`:
- **0xAA 0x55**:两字节固定帧头,标识帧起点,**不参与 CRC**;
- **LEN**:一字节,= TYPE(1 字节)+ payload 总长度(即 LEN = 1 + payload 长度),范围 1~64;
- **TYPE**:一字节,业务/命令类型;
- **payload**:变长,长度 = LEN − 1(LEN=1 时无 payload);
- **CRC**:两字节 CRC16,**小端**摆帧尾(先 CRC_LO,再 CRC_HI)。

最关键的一句:**CRC 的覆盖范围是 LEN + TYPE + payload 这三段,不含帧头、也不含 CRC 自己**——帧头是固定值、坏了在找帧头阶段就被丢,没必要进 CRC;CRC 字段自己当然不能自校验。

**Q2. CRC16-MODBUS 的参数?帧尾两字节怎么摆?上线前怎么保证算对?**

参数:多项式 0xA001(即 0x8005 的反射形式)、初值 0xFFFF、输入反射、输出反射、最后不做异或。算出的是个 16 位值,按**小端**写进帧尾——低 8 位放 CRC_LO、高 8 位放 CRC_HI。

保证算对靠**测试向量**:协议文档里给了几组"输入字节 → 期望 CRC"的标准向量(比如 `01 03 00 00 00 02` 应得 0x0BC4),发收两端的实现都得先把它们跑通,**任何一组对不上就是实现错了、绝不上线**。我就是用这几组向量把 STM32 侧和网关侧对齐的。

**Q3. TYPE 字典里关键的几类?为什么上下行要用不相交的号段?**

上行(STM32 → 网关):0x01 温湿度(payload = 温度×10 大端 2 字节 + 湿度×10 大端 2 字节 + 1 字节校验位,LEN=6)、0x02 光照(2 字节大端无符号,LEN=3)、0x03 心跳(无 payload,LEN=1)、0x05 查询应答(`[seq][结果码][数据]`)、0x06 ACK(`[seq][结果码]`)。下行(网关 → STM32):0x20 查光照、0x21 查温湿度、0x22 设采样周期(`[seq][周期 2 字节大端,单位秒]`)。

**上下行号段不相交**(上行 0x01–0x1F、下行 0x20–0x2F)是有意为之:TYPE 这一字节要穿过两端**同一套**解析代码,解出来的帧结构又不带方向信息。若上下行复用同一个号:一来同一个号得靠额外上下文才解释得清、违背"TYPE 自解释";二来一旦出错,一个错帧可能正好"伪装"成另一方向的合法帧被静默执行。分开号段后,收到不属于本方向的号就直接判非法丢弃,符合 Fail Fast。

**Q4. frame_build 怎么组帧?为什么它和接收端的 CRC 不可能算出不一样?**

组帧顺序:先写 0xAA、0x55,再写 LEN(= 1 + payload 长度)、TYPE、逐字节 payload;然后从 LEN 那个字节起、覆盖 LEN+TYPE+payload,用 crc16 从初值 0xFFFF 逐字节累加,最后把 16 位结果按小端写成 CRC_LO、CRC_HI。

之所以上行组帧、下行解析、网关侧校验三处的 CRC 必然一致,是因为它们**复用同一份 crc16_update**(同一个函数、同样的覆盖范围约定),不存在"各写各的、参数偷偷不一样"的可能——这也是我特意不另写一份 CRC、而是复用既有实现的原因。

**Q5. 这套请求-响应是怎么做到可靠的?(seq / 结果码 / 幂等 / 超时重试)**

四件套,缺一不可:
1. **序列号 seq**:每条下行命令的 payload 第一字节是 seq,STM32 应答时**原样带回**;网关靠它把收到的应答和"在途命令表"里的某条命令配对,从而知道"这是在回哪一条"。
2. **结果码**:应答里 seq 之后跟一字节结果码(0x00 成功 / 0x01 参数非法 / 0x02 不支持的命令 / 0x03 暂时不可用)。原则是 STM32 收到没法成功执行的命令**绝不静默丢、也不假装成功**,必须回带对应码的应答,让网关/运维据码定位。
3. **幂等**:STM32 记住最近处理过的 seq,若收到的 seq 与刚处理过的相同,判为"重发",**不重复执行命令本体、只把上次的应答再补发一次**。这样即便应答在路上丢了、网关重发,设置类命令也不会被执行两遍。
4. **超时重试契约**:网关发命令后规定 STM32 应在 500ms 内应答;超时没收到就用**同一个 seq**重发(配合上面的幂等),最多重试 3 次,仍无应答就判这条失败、记日志交上层处理。整体是"尽力而为",靠"seq + 有限重试 + 超时上报"这套轻量机制提供可靠性,而不在协议层做复杂的可靠重传状态机——匹配低速场景的简洁性目标。

**Q6. 这块怎么和项目一(网关)缝合成一条链路?**

靠**同一套 M5 协议**:STM32 这端(N8)是协议的**编码方**,把传感器数据和应答组成帧、从串口发出;树莓派网关那端(项目一 M5)是**解码方**,用状态机把字节流解回成帧、再落库/转发。一个发、一个收,共用同一份帧格式和同一份 CRC 实现约定,所以这两个项目不是两个孤立练习,而是一条完整的"端 → 边"数据链路。面试被问"你这两个嵌入式项目怎么连起来的",这就是答案。
