# 第一部分:纯面试(通用知识,最重要)

## 主题 A:GPIO 基础与配置(必考门槛)

### A1. 配置一个 GPIO(General-Purpose Input/Output,通用输入输出)引脚为输出,需要哪几步?顺序能不能乱?

**答题要点**(三步,顺序不能乱):
1. **开时钟**:先使能该 GPIO 端口所在的时钟(STM32F103 上 GPIO 挂在 APB2 总线,通过 RCC——Reset and Clock Control,复位与时钟控制——使能)。
2. **配模式**:配置该引脚的工作模式(输入/输出、推挽/开漏、速度)。
3. **读写数据**:通过数据寄存器读引脚电平或写输出电平。

**为什么顺序不能乱——这是最关键的考点**:**必须先开时钟,再配模式。** STM32 为省电,外设时钟默认是关的;时钟没开时,你对该端口配置寄存器的写操作**根本不生效**(寄存器没有时钟驱动)。很多人第一次点灯失败就是忘了开时钟,代码看着全对,灯就是不亮。

**措辞陷阱**:别只说"要开时钟"就完事,要主动点出"**不开时钟,后面的配置寄存器写了也白写**"——这个因果是面试官想听的,光知道"有这一步"不算理解。

---

### A2. STM32F103 配置一个 GPIO 输出,底层实际操作了哪几个寄存器?

**答题要点**(寄存器三步,对应 A1 的三步):
1. **RCC 寄存器**(如 `RCC_APB2ENR`):使能 GPIO 端口时钟。
2. **配置寄存器**(F103 是 `CRL`/`CRH`,Configuration Register Low/High):每个引脚 4 位,配 MODE(模式:输入还是输出 + 输出速度)+ CNF(配置:推挽/开漏/浮空/上拉下拉等)。低 8 个引脚在 CRL,高 8 个在 CRH。
3. **数据寄存器**:`IDR`(Input Data Register,输入数据寄存器)读输入;`ODR`(Output Data Register,输出数据寄存器)写输出;实际工程里写输出常用 `BSRR`(Bit Set/Reset Register,位设置/复位寄存器)。

**追问:为什么写输出常用 BSRR 而不直接写 ODR?**
因为 **BSRR 的写操作是原子的、且不影响其他位**。直接对 ODR 做"读-改-写"(读出整个寄存器、改某一位、写回)有竞态风险:如果中途有中断也改了同端口的别的位,会丢失。BSRR 让你"只置位/只清位某一引脚"一步到位,不碰其他位,无需读-改-写,天然安全。这是个能体现深度的点。

**怎么答出深度**:F103 用 `CRL`/`CRH` 是它特有的(每引脚 4 位、MODE+CNF 组合),而 F4/H7 系列改成了 `MODER`/`OTYPER`/`PUPDR` 等分开的寄存器。能说出"F103 是 CRL/CRH 把 MODE 和 CNF 挤在 4 位里"显出你真看过 F103 的寄存器。

### A3. 推挽输出(Push-Pull)和开漏输出(Open-Drain)有什么区别?各用在什么场合?

**答题要点**:
- **推挽输出**:能主动输出高电平也能主动输出低电平(内部上下两个管子,一个拉高一个拉低)。驱动能力强,电平干净。**适合驱动 LED、控制信号等普通数字输出。**
- **开漏输出**:只能主动拉低,输出高时是"高阻态"(浮空),必须**外接上拉电阻**才能呈现高电平。

**为什么需要开漏——关键场景**:
1. **电平转换**:芯片是 3.3V,要驱动 5V 逻辑,开漏 + 上拉到 5V 就能输出 5V 高电平,推挽做不到(推挽高电平被芯片供电钳在 3.3V)。
2. **线与(wired-AND)/ 总线共享**:多个设备接同一根线(典型如 I2C 总线),都用开漏 + 一个共同上拉。任一设备拉低线就是低,没人拉时上拉拉高。这样多设备共线不会"打架"(推挽的话一个输出高、一个输出低会短路烧管子)。

**怎么答出深度**:把"为什么 I2C 必须用开漏"主动带出来——I2C 是多主多从共享 SDA/SCL,必须线与,所以物理上要求开漏 + 上拉。这把 GPIO 模式和后面的 I2C 协议串起来了。

## 主题 B:GPIO 八种模式 + 输入(必考)

### B1. STM32F103 的 GPIO 有哪八种工作模式?

**答题要点**:四输入 + 四输出。

**四种输入模式**:
- **浮空输入**(Floating):引脚不接内部上拉/下拉,电平完全由外部决定。没外部信号时电平不确定(悬空)。
- **上拉输入**(Pull-up):内部接上拉,默认高电平。
- **下拉输入**(Pull-down):内部接下拉,默认低电平。
- **模拟输入**(Analog):关掉数字输入缓冲器,引脚直通 ADC(模数转换),给采集模拟电压用。

**四种输出/复用模式**:
- **推挽输出**(Push-Pull)
- **开漏输出**(Open-Drain)
- **复用推挽输出**(AF Push-Pull):引脚功能交给片上外设(如 USART 的 TX),推挽方式。
- **复用开漏输出**(AF Open-Drain):同上,开漏方式(如 I2C 的 SCL/SDA)。

**怎么答出深度**:别只列名,点出分类逻辑——输入按"内部要不要拉、是不是给 ADC"分四种;输出按"推挽/开漏 × 普通/复用"组合成四种。复用(AF,Alternate Function,复用功能)是关键:引脚被外设接管时就不是普通 GPIO 了。

**追问:按键接 GPIO,该用哪种输入模式?**
看按键电路。如果按键一端接 GND、按下接通到地,那应该用**上拉输入**:不按时内部上拉拉到高电平,按下拉到低——这样不按时电平确定(高),不会浮空乱跳。反过来按键接 VCC 就用下拉输入。**关键是不能用浮空输入**——浮空时不按的状态电平不确定,会乱读。

### B2. 什么是按键抖动?为什么要消抖?怎么消抖?

**答题要点**:
- **抖动**:机械按键按下/松开的瞬间,金属触点会有几毫秒的来回弹跳,导致电平在这几毫秒内高低反复跳变。如果直接读,一次物理按下会被误读成多次按下。
- **消抖**:确认电平稳定后才认定为一次有效按键。

**两类消抖方法**:
1. **软件消抖**:检测到电平变化后,延时一小段(如 10-20ms)再读一次,两次一致才确认。
2. **硬件消抖**:加 RC 滤波电路或施密特触发器,从电路上滤掉抖动。

**怎么答出深度——非阻塞消抖**:最朴素的软件消抖是 `delay(20ms)` 死等,但这会阻塞 CPU。更好的做法是**非阻塞消抖**:记录上次状态变化的时间戳,主循环每次跑过来比较"距上次变化是否超过消抖时间",超过了才确认——期间 CPU 不阻塞,能干别的。这对应嵌入式"不在主流程里 `delay` 死等"的通用思路。

**追问:为什么不用阻塞 delay?**
阻塞 delay 期间 CPU 什么都干不了——其他任务、其他按键、传感器采集全停了。在要同时干多件事的系统里,任何地方 `delay` 死等都是设计缺陷。非阻塞(时间戳比较)让单线程主循环能"并发"处理多件事。

## 主题 C:时钟与启动(高频)

### C1. STM32 上电后是怎么把主频跑到 72MHz 的?(F103 最高 72MHz)

**答题要点**:STM32 上电默认用内部 RC 振荡器(HSI,High Speed Internal,内部高速时钟,8MHz)启动;启动代码里的时钟配置会切到外部晶振(HSE,High Speed External,外部高速时钟)并经过 PLL(Phase-Locked Loop,锁相环)倍频。F103 典型配置:**HSE 8MHz × 9(PLL 倍频)= 72MHz**。

**怎么答出深度**:点出"为什么不直接用 HSI"——HSI 是内部 RC,精度差(温漂大),做串口波特率、USB 这种对时钟精度敏感的外设不够准;HSE 用外部晶振,精度高。所以高性能配置都切到 HSE+PLL。

**追问:各总线(AHB/APB1/APB2)的时钟一样吗?**
不一样。系统时钟(SYSCLK,72MHz)经过各级分频器分给 AHB(高速总线)、APB1(低速外设总线,F103 上限 36MHz)、APB2(高速外设总线,72MHz)。所以配置时要注意 APB1 上的外设时钟可能只有 36MHz,算波特率/定时器周期时不能直接用 72MHz。

## 主题 D:HAL 库(常被问到)

### D1. 什么是 HAL 库?用 HAL 和直接操作寄存器有什么区别?

**答题要点**:HAL(Hardware Abstraction Layer,硬件抽象层)是 ST 官方提供的、把寄存器操作封装成函数的库。比如 `HAL_GPIO_WritePin(GPIOC, GPIO_PIN_13, GPIO_PIN_RESET)` 底层就是操作 BSRR 寄存器。

**区别(权衡)**:
- **HAL 优点**:可移植(换芯片型号 API 基本不变)、开发快、配合 CubeMX 自动生成代码。
- **HAL 缺点**:有额外函数调用开销、生成代码体积大、对底层不透明(出问题难定位)。
- **寄存器操作优点**:最快、最省空间、完全可控。
- **寄存器操作缺点**:可移植性差、开发慢、易出错。

**怎么答出深度**:别站队"HAL 好"或"寄存器好",讲权衡——快速开发、跨型号用 HAL;对性能/体积/时序极致要求(如高频中断、bit-banging)用寄存器。能说"我用 HAL 开发,但理解它底层操作的是哪个寄存器"最好(对应 A2)。

**追问:HAL_Delay 是怎么实现的?用它要注意什么?**
`HAL_Delay` 基于 SysTick(系统滴答定时器,默认 1ms 中断一次,维护一个全局计数 tick)。`HAL_Delay(n)` 就是死等 tick 增加 n。**注意**:它是**阻塞**的;而且如果在**中断里调 `HAL_Delay`**,且该中断优先级 ≥ SysTick 中断优先级,会**死锁**——因为 SysTick 中断被你这个中断压着没法执行,tick 永远不增加,`HAL_Delay` 永远等不到。

# 第二部分:和项目挂钩(我具体怎么做的)

> 这部分是面试官追问"你自己实际做过吗"时用的细节支撑。

### P1. 你点灯那个 LED 是怎么接的?为什么写低电平才亮?

**答**:Blue Pill 板载 LED 在 **PC13**,接法是 **VCC → LED → PC13**(LED 正极接电源,负极接引脚)。所以 PC13 输出**低电平**时,引脚和 VCC 之间有压差,电流流过 LED 点亮;输出高电平时两端等电位,不亮。这叫**低电平点亮(低有效)**。所以我代码里 `HAL_GPIO_WritePin(GPIOC, GPIO_PIN_13, GPIO_PIN_RESET)`(写 0)是点亮。

**这能讲出的点**:不能想当然"写 1 就亮",要看硬件接法。低有效在嵌入式很常见(很多板子的 LED、复位都是低有效)。

---

### P2. 你的工程里时钟是怎么配的?

**答**:`SystemClock_Config()`(CubeMX 生成、我逐行看过)配的是 **HSE 8MHz 经 PLL ×9 = 72MHz** 作为系统时钟。GPIO 端口时钟在 `gpio.c` 里通过 `__HAL_RCC_GPIOC_CLK_ENABLE()` 使能(对应寄存器三步的第一步:开时钟),然后才配 PC13 为推挽输出。

**这能讲出的点**:我对照真实的 `gpio.c` 和 `main.c` 逐行拆过——知道 CubeMX 生成的 `MX_GPIO_Init` 里先调用 `__HAL_RCC_GPIOx_CLK_ENABLE`、再 `HAL_GPIO_Init` 填 `GPIO_InitTypeDef`(Pin/Mode/Pull/Speed),最后才能 WritePin。这印证了"先开时钟再配模式"那条铁律在生成代码里就是这个顺序。

---

### P3. 你做按键消抖用的是阻塞 delay 还是非阻塞?

**答**:N2 里我用的是**非阻塞消抖**——用 `HAL_GetTick()`(读 SysTick 维护的毫秒计数)记录上次电平变化的时间戳,主循环每轮比较"距上次变化是否超过消抖窗口",超过才确认这次按键。**没有用 `HAL_Delay` 死等**,因为死等会卡住主循环、其他事都干不了。

**这能讲出的点**:这是 D1 追问里"为什么不用阻塞 delay"的实际落地。我 N2 验收的三道题都答对了,核心就是这个非阻塞时间戳比较的思路。
