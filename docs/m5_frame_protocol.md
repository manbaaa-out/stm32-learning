## 1. 概述

### 1.1 目的

本协议定义 STM32 传感器节点(Project 2)与树莓派网关(Project 1)之间通过 UART 串口传输应用层数据的帧格式与解析规则。自 v1.1 起,协议支持**双向**:STM32 上行传感器数据,网关下行控制命令。

### 1.2 设计目标

按优先级排序:

1. **健壮性**:嵌入式现场存在电磁干扰,协议需检测并丢弃损坏帧,不能让接收端 FSM 卡死
2. **简洁性**:STM32 资源紧张(64KB Flash / 20KB RAM),协议实现不能过重
3. **可观测性**:解析错误、命令执行结果需可记录、可定位,运维友好
4. **效率**:慢速场景(≤ 10 Hz),字节开销不是瓶颈,可适当冗余换健壮性

### 1.3 设计哲学

本协议属于"**标记流派**" —— 用固定帧头标识帧边界,而非依赖时序(对比:Modbus RTU 属于"时序流派",靠 3.5 字符静默分帧)。

理由:UART 异步通信无严格时序约束,标记流派更通用;且接收端可以是 Linux 用户态程序,无法保证字符级时序精度。

### 1.4 基本约定

- **字节序**:多字节字段使用规则——帧级字段 CRC16 用**小端**(低字节在前);payload 内部的多字节业务字段用**大端**(网络字节序),上下行一致。
- **数值表示**:文档中所有十六进制值用 `0x` 前缀,如 `0xAA`
- **单帧最大长度**:69 字节(`2 + 1 + 64 + 2`)
- **单帧最小长度**:6 字节(心跳包:`2 + 1 + 1 + 2`)

### 1.5 方向定义(v1.1 新增)

| 术语 | 英文 | 方向 | 物理线 | 谁主动 | 典型内容 |
|---|---|---|---|---|---|
| **上行** | uplink | STM32 → 网关 | STM32 的 TX(PA9)→ 网关 RX | STM32 | 传感器数据、查询应答、ACK |
| **下行** | downlink | 网关 → STM32 | 网关 TX → STM32 的 RX(PA10) | 网关 | 控制命令(查询、设置) |

**关键原则:上行 TYPE 与下行 TYPE 使用不相交的数值段。**
物理上 TX/RX 是两根独立的线,帧不会在同一根线上混流;但 TYPE 字节要穿过两端**同一套**解析代码(`FrameParser`)和日志,且解析出的 `Frame{type,payload}` 结构体不携带方向信息。若上下行复用同一 TYPE,会导致:(a) 同一 TYPE 需外部上下文才能解释,违背 TYPE 自解释;(b) payload 格式按方向二义;(c) 故障时错帧可伪装成合法对帧、被静默执行。分段使错误"不在字典即丢弃",符合 Fail Fast。

## 2. 帧格式

### 2.1 完整帧结构

```
+--------+--------+------+------+------------------+--------+--------+
| 帧头1  | 帧头2  | LEN  | TYPE | payload data     | CRC_LO | CRC_HI |
+--------+--------+------+------+------------------+--------+--------+
  1 字节   1 字节   1B     1B     (LEN-1) 字节         1B       1B
```

帧结构**上行下行通用**,v1.1 未做任何改动。

### 2.2 字段表

| 字段 | 大小 | 取值范围 | 是否参与 CRC | 含义 |
|---|---|---|---|---|
| 帧头1 | 1 字节 | `0xAA`(固定) | ❌ 否 | 帧起始标识(高位 10101010) |
| 帧头2 | 1 字节 | `0x55`(固定) | ❌ 否 | 帧起始标识(高位 01010101) |
| LEN | 1 字节 | 1 ~ 64 | ✅ 是 | TYPE 字节 + payload data 字节的总长度 |
| TYPE | 1 字节 | 0x01 ~ 0xFE | ✅ 是 | 业务/命令数据类型标识 |
| payload data | (LEN - 1) 字节 | 任意 | ✅ 是 | 业务数据,格式由 TYPE 决定 |
| CRC_LO | 1 字节 | 0x00 ~ 0xFF | ❌ 否 | CRC16 低字节 |
| CRC_HI | 1 字节 | 0x00 ~ 0xFF | ❌ 否 | CRC16 高字节 |

### 2.3 CRC 覆盖范围

```
+--------+--------+------+------+------------------+--------+--------+
|  不含  |  不含  | LEN  | TYPE | payload data     |  不含  |  不含  |
+--------+--------+------+------+------------------+--------+--------+
                  └─────── CRC 计算范围 ──────────┘
```

**理由**:
- 帧头是固定值,损坏由 FSM 在头部识别阶段直接丢弃,不进入 CRC 计算
- CRC 字段本身不参与自校验
- 减少 CRC 计算量(嵌入式哲学:能省则省)

## 3. 字段详解

### 3.1 帧头(2 字节)

**固定值**:`0xAA 0x55`

**为什么是这个值**:二进制 `10101010 01010101`,位变化频繁,接收端 UART 时钟同步友好。这是工业标配(大疆 SDK、激光雷达、串口屏均使用)。

**为什么用 2 字节而不是 1 字节**:
- 1 字节帧头假阳性率 = 1/256 ≈ 0.39%(payload 中恰好出现帧头字节)
- 2 字节帧头假阳性率 = 1/65536 ≈ 0.0015%(降低 256 倍)
- 4 字节帧头更安全但过度设计,M5 场景不需要

### 3.2 LEN(1 字节)

**含义**:LEN = TYPE 字段长度(1B)+ payload data 长度

**合法取值**:1 ≤ LEN ≤ 64

**LEN 取值示例**:
- LEN = 1:心跳包(只有 TYPE,无 payload data)
- LEN = 2:状态包 / 查询命令(TYPE + 1 字节)
- LEN = 3:光照包 / ACK(TYPE + 2 字节)
- LEN = 6:DHT11 温湿度包(TYPE + 5 字节)

**LEN_MAX = 64 的推导**:
1. 列出业务上所有 TYPE 的 payload 长度,最大 33 字节(复合环境包 + 扩展余量)
2. 给 ~2 倍余量,向上取整到 2 的幂 → 64
3. 不选 128 是为了降低 LEN 损坏后落在合法范围的概率(防灾难传播)
4. 不选 32 是为了留扩展空间

**LEN_MAX 的可变性**:此值与业务挂钩,业务扩展时需重新评估。代码中以 `constexpr uint8_t LEN_MAX = 64;` 命名常量定义。

### 3.3 TYPE(1 字节)

**含义**:业务数据 / 命令类型标识

**TYPE 字典(v1.1)** —— 新增"方向"列:

| TYPE | 含义 | 方向 | payload 长度 | LEN |
|---|---|---|---|---|
| `0x00` | **保留(非法)** | — | — | — |
| `0x01` | DHT11 温湿度 | 上行 | 5 字节(温 2B + 湿 2B + 校验 1B) | 6 |
| `0x02` | BH1750 光照 | 上行 | 2 字节(光照值,大端 uint16) | 3 |
| `0x03` | 心跳包 | 上行 | 0 字节 | 1 |
| `0x04` | 设备状态(周期主动上报) | 上行 | 1 字节 bitmask(bit0=DHT11 OK, bit1=BH1750 OK, bit2~7 预留=0) | 2 |
| `0x05` | **查询应答**(回查询类命令,带状态/值) | 上行 | ≥ 2 字节(`[seq][结果码][数据…]`) | ≥ 3 |
| `0x06` | **ACK 命令确认**(回设置类命令,带结果码) | 上行 | 2 字节(`[seq][结果码]`) | 3 |
| `0x07` ~ `0x0F` | 上行预留 | 上行 | — | — |
| `0x10` | 复合环境包(规划中) | 上行 | ≤ 16 字节 | ≤ 17 |
| `0x11` ~ `0x1F` | 上行预留 | 上行 | — | — |
| `0x20` | **查询光照设备状态** | 下行 | 1 字节(`[seq]`) | 2 |
| `0x21` | **查询温湿度设备状态** | 下行 | 1 字节(`[seq]`) | 2 |
| `0x22` | **设采样周期** | 下行 | 3 字节(`[seq][周期 2B 大端,单位 ms]`) | 4 |
| `0x23` ~ `0x2F` | 下行命令预留 | 下行 | — | — |
| `0x30` ~ `0xEF` | 预留 | — | — | — |
| `0xF0` ~ `0xFE` | 厂商扩展预留 | — | — | — |
| `0xFF` | **保留(非法)** | — | — | — |

**段划分说明**:
- `0x01`~`0x1F`:上行段(数据 + 应答 + 预留)
- `0x20`~`0x2F`:下行命令段(16 个命令位)
- 上下行**不相交**,见 §1.5 原则。

**为什么 0x00 和 0xFF 保留**:这两个值是"全 0"和"全 1",最容易因电气问题(空线/短路)被误读出来,作为非法 TYPE 用于检测异常。

**TYPE 字段的校验责任**:协议层(M5 FrameParser)**不验证 TYPE 是否在字典中**,只保证帧的物理结构合法。TYPE 字典的合法性、方向的合法性(例如 STM32 收到一个上行 TYPE 应视为非法命令并回结果码 0x02)由**上层应用**检查。这是关注点分离(Separation of Concerns)—— 协议层稳定,业务层迭代不影响协议层。

### 3.4 payload data(变长)

**格式**:由 TYPE 决定,本协议层不规定。

**长度**:`LEN - 1` 字节。LEN = 1 时无 payload data。

**字节序约定**:payload 内部多字节业务字段使用**大端字节序**(网络字节序),**上下行一致**。

**示例**(DHT11 温湿度,TYPE = 0x01,上行):

```
payload data = [温度高字节] [温度低字节] [湿度高字节] [湿度低字节] [校验和]
                ↑─────── 温度 (×10) ───────↑ ↑──── 湿度 (×10) ────↑
                例:25.3°C → 0x00FD (= 253)   例:60.5% → 0x025D (= 605)
```

**示例**(设采样周期,TYPE = 0x22,下行):

```
payload data = [seq] [周期高字节] [周期低字节]
                ↑      ↑──── 周期 (ms, 大端 uint16) ────↑
                序列号  例:2000s → 0x07D0
```

**示例**(设备状态,TYPE = 0x04,上行):

​```
payload data = [状态 bitmask]
                ↑ bit0=DHT11 OK, bit1=BH1750 OK, bit2~7 预留=0
                例:温湿度正常 + 光照故障 → 0x01
                    两者都正常           → 0x03
​```

**重要**:此字节是**按位标志(bitmask)**,不是连续枚举值。接收端必须逐位 AND 判断各外设状态,不能把整字节当单一数值解释。合法取值仅 0x00 / 0x01 / 0x02 / 0x03。

## 4. 校验算法

### 4.1 算法参数

| 参数 | 值 |
|---|---|
| 算法 | CRC16-MODBUS |
| 多项式 | `0xA001`(= `0x8005` 反射) |
| 初始值 | `0xFFFF` |
| 反射输入 | 是 |
| 反射输出 | 是 |
| 末位异或 | 无(XOR 0x0000) |

**为什么选 CRC16-MODBUS**:嵌入式串口事实标准;对 ≤ 1KB 帧检测能力优秀;计算开销小(查表法每字节 ~10 个时钟周期)。

### 4.2 算法实现

推荐使用**查找表法**(空间换时间,256 项 × 2 字节 = 512 字节查找表):

```c
static const uint16_t kCrc16Table[256] = { /* 256 项预算值 */ };

uint16_t crc16_update(uint16_t crc, uint8_t byte) {
    return (crc >> 8) ^ kCrc16Table[(crc ^ byte) & 0xFF];
}
```

### 4.3 测试向量(必须通过)

发送/接收两端必须使用以下测试向量验证 CRC 实现一致性。**只要任一向量不匹配,实现错误,绝不能上线**。

| 输入字节(hex) | 期望 CRC16(uint16_t) | CRC_LO | CRC_HI |
|---|---|---|---|
| `01 03 00 00 00 02` | `0x0BC4` | `0xC4` | `0x0B` |
| `01 06 00 01 00 03` | `0x0B98` | `0x98` | `0x0B` |
| `02 01 00 00 00 02` | `0xF8BD` | `0xBD` | `0xF8` |
| `04` | `0x83BE` | `0xBE` | `0x83` |

**外部验证渠道**:可用 https://crccalc.com/ 选 "CRC-16/MODBUS" 算法比对(注意网站字节顺序显示,需做字节交换比对)。

### 4.4 在帧中的字节顺序

CRC16 的 16 位值按**小端**写入帧尾:

```
计算得 CRC = 0x1234
帧尾两字节 = [0x34] [0x12]
              CRC_LO  CRC_HI
```

**接收端拼接公式**:
```c
uint16_t received = ((uint16_t)crc_hi << 8) | crc_lo;
```

## 5. 错误处理

### 5.1 错误类型与处理对照表

| 错误类型 | 发生时机(FSM 状态) | 处理动作 | 数据损失 |
|---|---|---|---|
| 收到非 `0xAA` 字节 | WAIT_HEADER1 | 静默丢弃,留在当前状态 | 0 字节 |
| 收到非 `0x55` 非 `0xAA` 字节 | WAIT_HEADER2 | 回 WAIT_HEADER1 | 0 字节 |
| 收到 `0xAA` 字节 | WAIT_HEADER2 | 自环,留在当前状态 | 0 字节 |
| LEN = 0 或 LEN > 64 | WAIT_LEN | 记 ERROR,resync 到 WAIT_HEADER1 | 1 帧 |
| payload 数据损坏 | READ_PAYLOAD | (不立即检测,等 CRC 阶段) | — |
| CRC 不匹配 | DELIVER | 记 ERROR,丢弃整帧,resync | 1 帧 |
| 收到 unexpected 字节 | DELIVER(瞬态) | 记 ERROR,resync | 1 帧 |

### 5.2 resync 机制

**核心思想**:任何错误路径都汇聚到一个安全起点(WAIT_HEADER1)。**最坏代价**:丢 1 帧。下一帧能正常解析,FSM 永不卡死。

### 5.3 LEN 合理性检查(防灾难传播)

WAIT_LEN 状态收到字节后**立即检查范围**:

```c
if (len < 1 || len > 64) {
    log_error("LEN out of range");
    state = WAIT_HEADER1;  // resync
    return;
}
```

早期检查把灾难影响压缩到只丢 1 帧,体现 **Fail Fast** 原则。

### 5.4 日志规范

错误日志应包含足够上下文,**让运维不读代码即可定位问题**:

```
[ERROR] LEN out of range: 80, expected [1, 64]
[ERROR] CRC mismatch: computed=0x5CA0, received=0xFF5B
```

## 6. 请求-响应模型(v1.1 新增)

本章是 v1.1 的核心增量,定义**下行命令与上行应答如何配对、如何表达结果、链路抖动时如何约定行为**。注意:协议层只定义**契约**,不规定网关/STM32 的具体实现(定时器、线程模型等属于实现策略)。

### 6.1 为什么需要请求-响应模型

v1.0 是单向"播报":STM32 主动发,网关被动收,无需配对。v1.1 引入下行命令后,出现"一问一答"语义:

- 网关发"查询光照(0x20)" → 期望 STM32 回"查询应答(0x05)"
- 网关发"设采样周期(0x22)" → 期望 STM32 回"ACK 确认(0x06)"

串口异步、可能乱序、可能丢帧、可能重发。**没有配对机制,网关无法判断收到的应答对应哪条命令**,也无法处理重发导致的重复执行。

### 6.2 序列号 seq(命令-应答配对的命脉)

**定义**:每条**下行命令**的 payload 第一个字节是 `seq`(序列号 / 请求 ID,1 字节,0x00~0xFF 循环)。STM32 应答时**原样带回**同一个 seq。

**取值规则**:
- 网关为每条新命令分配递增 seq(到 0xFF 后回绕到 0x00)。
- **重发同一条命令时,必须复用原 seq**(不分配新值)。

**配对**:网关收到应答后,用应答里的 seq 匹配自己"在途命令表"里的某条命令,从而确定"这是在回哪一条"。

**幂等**:STM32 应记住最近处理过的 seq(至少最近一个)。若收到 seq 与刚处理过的相同,判定为"重发",**不重复执行命令本体,只补发一次应答/ACK**。这样即便 ACK 丢失、网关重发,设置类命令也不会被执行两次。

### 6.3 结果码字典(让失败可定位)

**定义**:上行应答(0x05 查询应答 / 0x06 ACK)的 payload 中,seq 之后紧跟 1 字节**结果码**。

| 结果码 | 含义 | 说明 |
|---|---|---|
| `0x00` | 成功 | 命令已正确执行 |
| `0x01` | 参数非法 / 超范围 | 如设采样周期 = 0 |
| `0x02` | 不支持的命令 | TYPE 不在下行命令字典(含收到上行 TYPE) |
| `0x03` | 设备忙 / 暂时无法执行 | 资源占用,可稍后重试 |
| `0x04` ~ `0xFF` | 预留 | — |

**原则**:STM32 收到无法成功执行的命令,**绝不静默丢弃,也不假装成功**,必须回一个携带对应结果码的应答。运维据结果码即可定位(对应 §5.4 哲学)。

### 6.4 应答类型的选择(0x05 vs 0x06)

| 命令类别 | 示例 | 期望应答 | 应答携带 |
|---|---|---|---|
| **查询类** | 0x20 查光照、0x21 查温湿度 | 0x05 查询应答 | seq + 结果码 + **数据** |
| **设置类** | 0x22 设采样周期 | 0x06 ACK 确认 | seq + 结果码(**无数据**) |

分开的理由:查询要的是"内容",设置要的是"回执"。压成一个 TYPE 会导致 payload 格式按命令类别二义。

### 6.5 超时与重试契约(只定契约,不定实现)

协议层仅约定以下**契约**,具体定时器/重试逻辑由网关实现自行决定:

1. **应答时限**:STM32 收到合法命令后,应在 **500 ms** 内回应(查询应答或 ACK)。该值为契约上限,STM32 实现应远快于此。
2. **重发用同 seq**:网关在超时未收到应答时若重发,**必须复用原命令的 seq**(见 §6.2 幂等)。
3. **建议重试次数**:网关对单条命令建议最多重试 **3 次**;仍无应答则判定命令失败,记 ERROR 日志,交由上层(MQTT/运维)处理。
4. **不保证投递**:本协议为**尽力而为**(best-effort),不在协议层做可靠重传的复杂状态机;可靠性由"seq + 有限重试 + 超时失败上报"这套轻量机制提供,符合 ≤10Hz 慢速场景的简洁性目标。

## 7. 完整帧示例

> 下行/上行新帧的 CRC 由 CRC16-MODBUS 实算(与 `experiments/m5_parser/test_crc16.cpp` 同源算法,已交叉自检 v1.0 已知向量一致)。

### 7.1 上行:心跳包(LEN=1,v1.0)

```
完整帧:    AA 55 01 03 40 21
CRC 计算:  CRC16({01, 03}) = 0x2140  →  CRC_LO=0x40, CRC_HI=0x21
```

### 7.2 上行:BH1750 光照帧(LEN=3,v1.0)

```
完整帧:    AA 55 03 02 01 90 A0 5C
CRC 计算:  CRC16({03, 02, 01, 90}) = 0x5CA0  →  CRC_LO=0xA0, CRC_HI=0x5C
```

### 7.3 下行:查询光照设备状态(0x20,seq=7)

```
完整帧:    AA 55 02 20 07 88 02
            │   │   │  │  │  └CRC┘
            └帧头┘   │  │  └ payload = [seq=0x07]
                    │  └ TYPE=0x20 查询光照
                    └ LEN=2 (TYPE + 1 字节 seq)
CRC 计算:  CRC16({02, 20, 07}) = 0x0288  →  CRC_LO=0x88, CRC_HI=0x02
帧长:     7 字节
```

### 7.4 下行:查询温湿度设备状态(0x21,seq=8)

```
完整帧:    AA 55 02 21 08 C9 96
CRC 计算:  CRC16({02, 21, 08}) = 0x96C9  →  CRC_LO=0xC9, CRC_HI=0x96
帧长:     7 字节
```

### 7.5 下行:设采样周期(0x22,seq=9,周期=2000ms)

```
完整帧:    AA 55 04 22 09 07 D0 0C 16
            │   │   │  │  │  └─┬─┘ └CRC┘
            └帧头┘   │  │  │   └ 周期 = 0x07D0 = 2000 (ms, 大端)
                    │  │  └ seq = 0x09
                    │  └ TYPE=0x22 设采样周期
                    └ LEN=4 (TYPE + seq + 2字节周期)
CRC 计算:  CRC16({04, 22, 09, 07, D0}) = 0x160C  →  CRC_LO=0x0C, CRC_HI=0x16
帧长:     9 字节
```

### 7.6 上行:查询应答(0x05,回 0x20,seq=7,成功,光照=400lux)

```
完整帧:    AA 55 05 05 07 00 01 90 CC C6
            │   │   │  │  │  │  └─┬─┘ └CRC┘
            └帧头┘   │  │  │  │   └ 光照值 = 0x0190 = 400 (大端 uint16)
                    │  │  │  └ 结果码 = 0x00 成功
                    │  │  └ seq = 0x07 (原样带回命令的 seq)
                    │  └ TYPE=0x05 查询应答
                    └ LEN=5 (TYPE + seq + 结果码 + 2字节数据)
CRC 计算:  CRC16({05, 05, 07, 00, 01, 90}) = 0xC6CC  →  CRC_LO=0xCC, CRC_HI=0xC6
帧长:     10 字节
注意:第一个 05 是 LEN,第二个 05 是 TYPE,数值巧合相同。
```

### 7.7 上行:ACK 确认(0x06,回 0x22,seq=9,成功)

```
完整帧:    AA 55 03 06 09 00 E6 31
            │   │   │  │  │  │  └CRC┘
            └帧头┘   │  │  │  └ 结果码 = 0x00 成功
                    │  │  └ seq = 0x09
                    │  └ TYPE=0x06 ACK 确认
                    └ LEN=3 (TYPE + seq + 结果码)
CRC 计算:  CRC16({03, 06, 09, 00}) = 0x31E6  →  CRC_LO=0xE6, CRC_HI=0x31
帧长:     8 字节
```

### 7.8 上行:ACK 确认(0x06,回 0x22,seq=9,参数非法)

```
完整帧:    AA 55 03 06 09 01 27 F1
            结果码 = 0x01 参数非法(如周期设为 0)
CRC 计算:  CRC16({03, 06, 09, 01}) = 0xF127  →  CRC_LO=0x27, CRC_HI=0xF1
帧长:     8 字节
```

 │ ────────────────────────────────>│   STM32 见 seq=9 已处理→不重复执行,
 │                                   │   只补发 ACK
 │  AA 55 03 06 09 00 E6 31          │
 │ <────────────────────────────────│

## 9. 版本演进计划

| 版本 | 内容 |
|---|---|
| v1.0 | 基础协议 + 5 个上行 TYPE(0x01-0x04) |
| **v1.1(当前)** | **双向化:下行命令段(0x20-0x2F)+ 上行应答(0x05/0x06)+ 请求-响应模型(seq/结果码/超时重试契约);向后兼容 v1.0** |
| v2.0(规划) | 协议版本号字段 / 时间戳字段 / 多设备寻址(需协议头扩展,不向后兼容) |

### 9.1 v1.1 明确不做的(避免过度设计)

以下为真实工业协议常见、但对**当前单 STM32 + 单网关点对点**场景属过度设计,明确推迟:

- **协议版本号字段**:加它会破坏现有帧结构、需改 FSM;当前上下行就一对设备,版本不会错配。
- **时间戳字段**:STM32 无 RTC,其时间戳只是开机计时,意义有限;真正时间戳由网关收到时打(`time(nullptr)`)已足够。
- **多设备寻址(地址字段)**:点对点 UART 非 RS-485 一对多总线,地址字段纯属浪费字节。

> 主动声明"考虑过但当前不做",是工业级判断:既知道协议能长成什么样,也知道现在该长到哪。

## 附录 A:与其他协议对比

| 维度 | Modbus RTU | DJI SDK | M5(本协议) |
|---|---|---|---|
| 分帧机制 | 时间静默(T3.5) | 帧头 0xAA | 帧头 0xAA 0x55 |
| 长度字段 | 隐含(功能码) | 13 字节头部 | 1 字节 LEN |
| 校验 | CRC16-MODBUS | CRC16+CRC32 | CRC16-MODBUS |
| 双向命令 | 主从读写 | 有 | 上行数据 + 下行命令(seq 配对) |
| 适用 | RS-485 主从总线 | 高复杂度 SDK | 嵌入式点对点 UART |
