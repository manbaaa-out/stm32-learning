# N9 独立看门狗(IWDG)速记卡

> 适用：项目二 STM32+FreeRTOS 传感器节点 / 嵌入式面试

## 卡 1 — 看门狗解决什么问题（为什么需要它）

**Part 1 · 面试通用**

Q：单片机已经有 HardFault 等异常处理了，为什么还需要看门狗？

A：异常处理只能覆盖"程序跑飞、触发硬件异常"这一类显式崩溃。它覆盖不了另一类更隐蔽的故障——程序"假死"：CPU 仍在正常取指执行、没有触发任何异常，但因为死循环、外设半挂死导致的无限阻塞、或任务间死锁，程序实际上已经不再推进有效工作。这种状态下芯片"活着但不干活"，异常机制完全无感。

看门狗（Watchdog）是针对"假死"的最后一道防线：它是一个独立于 CPU 执行流的硬件倒计时计数器，程序必须周期性地将其重置（行话称"喂狗"，refresh）；一旦在设定时间内未被喂狗，硬件强制复位整个芯片。其核心价值在于——它不依赖 CPU 仍在正确执行，因此能在 CPU 逻辑已经失效时仍然生效。

补充关键点：HardFault 默认进入死循环（`while(1)`）后并不会自我恢复，本质上也是"活着但不干活"。因此看门狗不仅兜底逻辑假死，也兜底"崩溃后卡在 HardFault"的场景。

**Part 2 · 本项目**

本项目是无人值守的现场传感器节点，没有人在旁边按复位键。代码中存在多处无限阻塞假设：`xStreamBufferReceive(..., portMAX_DELAY)`、`HAL_UART_Transmit(..., HAL_MAX_DELAY)`、以及 DHT11/BH1750 读取中的忙等。这些都建立在"外设和数据必然按预期到来"的前提上。一旦传感器接触不良半挂死（N4/N5 阶段已多次踩到杜邦线接触问题）或 I2C 总线被拉死，相关任务会卡在阻塞点出不来，喂给网关的心跳随之停止。IWDG 是该节点实现自恢复（卡死即自动重启）的手段。

## 卡 2 — IWDG 为什么"独立"：时钟源

**Part 1 · 面试通用**

Q：IWDG 的"独立"体现在哪里？为什么这个设计很重要？

A："独立"的核心是时钟源独立。IWDG 由专用的低速内部 RC 振荡器 LSI（约 40kHz）驱动，不使用 CPU 主时钟。

这个设计是其可靠性的根基。设想一种故障：时钟配置出错或主晶振失效，导致主时钟整体停摆。如果看门狗也依赖主时钟，它会随主时钟一起停止——在最需要它救援的时刻自身也失效了。挂在独立的 LSI 上，保证了主时钟停摆时它不停、CPU 卡死时它照常倒计时。

代价：LSI 是 RC 振荡器，频率随温度、电压漂移（标称 40kHz 但实测会偏），因此 IWDG 的超时时间不精确。但救命用途不要求精确——只需保证"卡死超过约 N 秒即复位"，N 是 2.0 还是 2.3 秒不影响目的。结论上要为这种漂移预留余量。

**Part 2 · 本项目**

`stm32f1xx_hal_conf.h` 中 `#define LSI_VALUE 40000U`，注释明确写有 "real value may vary depending on variations in voltage and temperature"，即官方承认 LSI 漂移。本项目超时按标称 2 秒配置，但喂狗周期取得明显更短以吸收漂移（见卡 4）。

## 卡 3 — 超时时间计算（PR / RLR / 手算）

**Part 1 · 面试通用**

Q：IWDG 的超时时间由什么决定？怎么算？

A：由三个寄存器配合，关键是两个：
- PR（Prescaler Register，预分频寄存器）：对 40kHz 的 LSI 分频，可选系数 4/8/16/32/64/128/256。
- RLR（Reload Register，重装载寄存器）：12 位，计数器初始值，最大 4095。喂狗即把计数器重新装回 RLR 值。
- （KR，Key Register，键值寄存器：写特定键值用于解锁配置 / 启动 / 喂狗，是操作 IWDG 的总开关。）

计算公式：

```
每个 tick 时间 = 预分频系数 / LSI
超时时间      = (RLR + 1) × 预分频系数 / LSI
```

正向验算（预分频 64，RLR 1250）：
```
tick   = 64 / 40000 = 1.6 ms
超时   = (1250 + 1) × 1.6 ms ≈ 2002 ms ≈ 2 s
```

反向求 RLR（要 2 秒、预分频 64）：
```
RLR = 超时(s) × LSI / 预分频 − 1
    = 2 × 40000 / 64 − 1 = 1249 ≈ 1250
```

选型经验：预分频越小，tick 越快，RLR 满值能撑的时间越短（预分频 4 时满值仅约 0.4 秒，做不出秒级超时）；预分频越大可拉长超时但分辨率变粗。秒级超时常用 64~256 区间。

**Part 2 · 本项目**

CubeMX 配置：IWDG counter clock prescaler = 64，down-counter reload value = 1250，对应超时 ≈ 2.0016 秒。

## 卡 4 — 超时时间 vs 喂狗周期：防误杀

**Part 1 · 面试通用**

Q：喂狗周期应该怎么定？为什么不能贴着超时时间设？

A：两个时间必须区分：
- 超时时间 T_wdg：多久没喂就复位。
- 喂狗周期 T_feed：代码多久喂一次狗。

铁律：T_feed 必须**远小于** T_wdg，而不是"小于即可"。原因有二：
1. LSI 漂移会让真实超时比标称值缩短，若喂狗周期贴得太近，正常程序也可能在缩短后的超时内来不及喂狗。
2. 任务可能被高优先级抢占而短暂延迟，喂狗时刻随之后移。

二者贴太近的后果是"误杀"——程序明明健康，却因喂狗周期与超时时间间距不足而被冤枉复位。

经验法则：T_feed 取 T_wdg 的 1/2 ~ 1/3。超时 2 秒，喂狗周期取 500ms~1s 较稳。

**Part 2 · 本项目**

`vSampleReportTask` 以 `HEARTBEAT_MS = 1000` 的绝对栅格（`vTaskDelayUntil`）每 1 秒轮转一次，天然是喂狗时基。1s 喂狗 / 2s 超时 = 1/2，余量合适，无需额外加定时器。

## 卡 5 — 三种"重启"的本质区别（高频追问）

**Part 1 · 面试通用**

Q：IWDG 复位、按物理复位键（NRST）、HardFault 进死循环，三者本质区别是什么？

A：按"芯片是否发生真正的系统复位"来切分：

- IWDG 复位 与 NRST 物理复位：本质相同，都触发系统复位（System Reset）。CPU 核、外设、寄存器全部回到上电初始状态，程序从向量表复位入口重新执行（等同重跑 SystemInit → main）。复位"深度"一致，区别仅在触发源——一个由 IWDG 计数器到 0 内部触发，一个由 NRST 引脚外部触发。

- HardFault 进死循环：根本没有发生复位。程序跑飞进入 HardFault_Handler 后卡在 `while(1)`，寄存器未清、外设状态保留、CPU 在死循环中空耗。结果上等同"假死"——活着但不干活，不会自我恢复，同样需要看门狗兜底。

进阶补充：可在 HardFault_Handler 中主动调用 `NVIC_SystemReset()` 立即重启，而非等待看门狗超时，缩短恢复延迟。

## 卡 6 — 看门狗监控的是什么（最易误判的概念）

**Part 1 · 面试通用**

Q：看门狗能监控到每个任务是否健康吗？

A：不能。看门狗本身是一个"傻"计数器，不感知系统里有几个任务、哪个该活哪个该死，它只认一件事——有没有人在喂它。因此它监控的不是"任务健康度"，而是"喂狗那行代码所在的执行路径是否仍可达"。喂狗点覆盖不到的执行路径，即为监控盲区。

由此引出多任务下的核心陷阱：若让每个任务各自独立喂狗，则只要任意一个任务还活着在喂，计数器就被持续重置，看门狗永不复位。这等于把监控逻辑从"所有任务都健康（逻辑与）"偷偷降级为"至少有一个任务还活着（逻辑或）"——死掉的任务被健康任务的喂狗动作所掩护。这是灾难性的降级。

须澄清的常见错误：整个芯片只有一个 IWDG、一个计数器；多任务调用喂狗喂的是同一个计数器，不存在"每任务一个计数器"。该计数器到 0 的充要条件是——所有喂狗点全部停止、无任何一个仍在喂。

正确做法：设统一的"集合点"，由单一角色负责喂狗，但喂狗前须确认所有被监控任务近期都已"报到"；任一任务该报到却未报到，则故意不喂、放任超时复位。这样"一次喂狗"才真正承载"所有任务都健康"的语义，把逻辑或扳回逻辑与。

## 卡 7 — 周期型任务 vs 事件型任务：监控原则

**Part 1 · 面试通用**

Q：一个事件驱动的任务平时一直阻塞等待事件、长期不动，怎么纳入看门狗监控？

A：先确立根本原则——**看门狗要监控的是"任务该动的时候有没有动"，而不是"任务有没有一直在动"。**

据此区分两类任务：
- 周期型任务：本就该周期性执行（如固定节拍的采样上报）。"一个周期内未动"即异常，直接用打卡监控成立。
- 事件型任务：仅在事件到来时才执行（如收到命令才处理），"长期不动"是正常状态。若强行要求它"必须定期动"，会把健康的空闲任务误判为死亡——典型误杀。

事件型任务的纳入手法（由糙到精）：
1. 不纳入：只监控周期型骨架任务。简单，但事件型任务存在监控盲区。复杂度低的项目可接受。
2. 给阻塞设上限：将无限阻塞改为有限超时，使任务即使无事件也能周期性"醒来打卡"，从而把事件型伪装成周期型。最常用。代价是极小的空转唤醒开销。
3. 区分"存活"与"卡死"两层：仅在真正干活的执行路径上设时限监控（如"开始处理一帧→处理完毕"不超过 X ms），空闲等待不计入。最精细，一般项目用不到。

关键实现细节：以方法 2 为例，打卡动作必须放在"无论是否取到事件都会执行到"的位置（即超时返回与正常返回都经过的路径）。若把打卡误放进"取到事件才进入"的分支内，则退回事件型陷阱——只有干过活才打卡，没事件就报不了到。

**Part 2 · 本项目**

三任务分类与处理：
- `vSampleReportTask`：周期型（1s 节拍），系统心跳骨架，必须打卡。
- `vCmdTask`：事件型，原 `xStreamBufferReceive(..., portMAX_DELAY)` 改为 `pdMS_TO_TICKS(500)`，超时返回后无论 n 是否为 0 都打卡。
- `vTxTask`：事件型，原 `xQueueReceive(..., portMAX_DELAY)` 改为 `pdMS_TO_TICKS(500)`，打卡放在 `if (xQueueReceive...)` 之外（取到则发送，未取到不发送，但任务存活照常打卡）。

500ms 唤醒周期的依据：判定喂狗的节拍是 1 秒，要求两事件型任务在每个 1 秒判定窗口内都已报到；取 500ms 留出余量，最坏情况下每 500ms 报到一次，稳落在窗口内。

## 卡 8 — 全员报到机制的实现：事件组

**Part 1 · 面试通用**

Q：FreeRTOS 下如何实现"所有任务都报到才喂狗"？为什么用事件组而不是普通标志变量？

A：核心数据结构是一组"报到标志位"，每个被监控任务占一个 bit。普通全局变量的置位操作 `flags |= BIT` 是"读—改—写"三步，多任务并发下存在竞态。FreeRTOS 的事件组（Event Group，事件标志组）为"多 bit 等全齐"场景而设计，其置位/查询操作是任务安全的，是该场景的标准解法。

核心 API：
- `xEventGroupCreate()`：创建事件组，返回句柄；内部为一组（24 位）标志位。依赖动态分配。
- `xEventGroupSetBits(group, bits)`：任务上下文置位，并发安全。任务用它给自己的 bit 打卡。
- `xEventGroupGetBits(group)`：读出当前全部 bit。
- `xEventGroupClearBits(group, bits)`：清位，开始新一轮收集。

一个重要取舍：此处用 `GetBits + 判断 + 主动决定喂不喂`，而不用 `xEventGroupWaitBits` 阻塞等全齐。因为我们要的语义是"没齐就放任超时复位"，而非"没齐就一直等"——后者会让监控点睡死，反而绕过了看门狗。

时序陷阱：判定喂狗的任务自身也要打卡，且必须"先打卡、再判定"。若先判定，则其自身 bit 可能尚未置上（上一轮清零后还没补打），导致永远凑不齐。

判定逻辑骨架（由 1 秒节拍任务承担）：
```c
for (;;) {
    xEventGroupSetBits(g_wdg_events, WDG_BIT_SAMPLE);          // ① 自己先打卡
    EventBits_t bits = xEventGroupGetBits(g_wdg_events);
    if ((bits & WDG_BIT_ALL) == WDG_BIT_ALL) {                 // ② 全齐？
        HAL_IWDG_Refresh(&hiwdg);                              //    齐 → 喂狗
        xEventGroupClearBits(g_wdg_events, WDG_BIT_ALL);       //    清零，开新一轮
    }                                                          // 没齐 → 不喂，放任倒计时
    /* ... 心跳 / 采样 / vTaskDelayUntil 1s ... */
}
```

**Part 2 · 本项目**

- bit 定义：`WDG_BIT_SAMPLE = 1<<0`、`WDG_BIT_CMD = 1<<1`、`WDG_BIT_TX = 1<<2`、`WDG_BIT_ALL = 三者或`。
- 句柄：`static EventGroupHandle_t g_wdg_events;`，在 main 的 USER CODE BEGIN 2 中 `xEventGroupCreate()` + 判 NULL + `Error_Handler()`（与 `xTxQueue` 等一致的失败即停策略）。
- 头文件：`event_groups.h`（非 task.h）。
- 配置前提：`configSUPPORT_DYNAMIC_ALLOCATION = 1`、`configUSE_TIMERS = 1` 已满足，事件组依赖此二者。

## 卡 9 — API 与 CubeMX 落地

**Part 1 · 面试通用**

Q：HAL 层操作 IWDG 用到哪些函数？

A：极简，两个：
- `MX_IWDG_Init()`（CubeMX 生成）：配好预分频/重装载值，并通过 `HAL_IWDG_Init(&hiwdg)` 启动 IWDG。其返回那一刻倒计时即开始。
- `HAL_IWDG_Refresh(&hiwdg)`：喂狗。一次调用即把计数器重装回 RLR 值（内部向 KR 写入键值 0xAAAA 触发重装载）。这是运行期唯一需主动调用的函数。

配置交给 CubeMX，运行期仅需周期性调用 Refresh。

**Part 2 · 本项目**

CubeMX 操作要点：
- System Core → IWDG → 勾选 Activated；Parameter Settings 填 prescaler=64、reload=1250。
- 生成后须确认三处：① `stm32f1xx_hal_conf.h` 中 `HAL_IWDG_MODULE_ENABLED` 由注释变为启用；② 新增 `iwdg.c/iwdg.h`，含 `IWDG_HandleTypeDef hiwdg`；③ `MX_IWDG_Init()` 被插入 main 初始化序列（I2C 之后、USER CODE BEGIN 2 之前）。

## 卡 10 — IWDG 的两个工程陷阱

**Part 1 · 面试通用**

Q：使用 IWDG 时有哪些容易翻车的点？

A：两个典型陷阱。

1. 开机即复位。IWDG 一旦由 `MX_IWDG_Init()` 启动便无法停止（除非复位），且从启动那刻即开始倒计时。从启动到调度器运行、首个任务首次喂狗之间存在一段时间，包含队列/信号量/任务创建、调度器启动、首轮采样等。若这段时间超过设定超时，芯片会在正式开工前被复位，陷入"开机即复位"死循环。对策：超时不设过短，并让首次喂狗尽早发生（如喂狗点置于周期任务循环体顶部，作为"本轮确已开始执行"的最早证据）。本项目超时 2 秒、初始化仅数毫秒，余量充足，但须理解该因果。

2.（手动移植 FreeRTOS 工程特有）异常向量重复定义。详见卡 11。

## 卡 11 — CubeMX × FreeRTOS 手动移植：三向量冲突（项目实测）

**Part 1 · 面试通用**

Q：在手动移植 FreeRTOS 的 CubeMX 工程里，重新生成代码后出现 `SVC_Handler / PendSV_Handler / SysTick_Handler` multiple definition，原因和处理？

A：FreeRTOS 的上下文切换依赖三个异常处理，实现全在 port.c 中：
- SVC_Handler：启动第一个任务（SVC 异常中恢复首个任务上下文）。
- PendSV_Handler：执行任务切换（FreeRTOS 故意将切换置于最低优先级的 PendSV，避免打断中断）。
- SysTick_Handler：系统节拍，每 tick 推动调度与延时。

FreeRTOSConfig.h 通过宏将 port.c 中的函数改名为向量表期望的标准名：
```c
#define vPortSVCHandler     SVC_Handler
#define xPortPendSVHandler  PendSV_Handler
#define xPortSysTickHandler SysTick_Handler
```
而 CubeMX 生成 `stm32f1xx_it.c` 时不感知 FreeRTOS，会按惯例再生成一份同名空桩，导致同一符号两份定义、链接冲突。

处理：在 it.c 中将这三个 handler 关闭（`#if 0` 包裹或删除），由 port.c 独占（DebugMon_Handler 保留，FreeRTOS 不接管）。

根因与长效对策：根因是采用了"手动移植"而非"CubeMX FreeRTOS 中间件"——后者生成的 it.c 会自动省略这三个 handler。手动移植下每次重新生成代码都会复发，须再次关闭。建议在关闭处写明醒目注释，把踩坑经验固化进代码本身，而非依赖人工记忆每次手动处理。

**Part 2 · 本项目**

本次为加 IWDG 而重新生成代码即触发此冲突。已用 `#if 0` 关闭 it.c 的三个 handler，DebugMon_Handler 保留。最终：0 warning、链接通过、RAM ≈ 62% / FLASH ≈ 62%。

## 一句话因果链（背诵用）

现场无人值守 → 程序可能假死（不崩溃但不干活，HardFault 死循环亦然）→ 需不依赖 CPU 的硬件自救 → IWDG 倒计时到 0 强制系统复位 → 挂在独立 LSI 上，主时钟死它不死 → 超时由 PR/RLR 算出（秒级用预分频 64~256），喂狗周期取超时 1/2~1/3 防误杀 → 多任务不可各喂各的（会退化为逻辑或），须"全员报到才喂"扳回逻辑与 → 事件组实现报到、先打卡再判定 → 事件型任务用有限超时唤醒打卡纳入监控 → 警惕开机即复位与手动移植的三向量冲突。
